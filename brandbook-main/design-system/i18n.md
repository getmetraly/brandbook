# Internationalisation (i18n)

Supporting multiple languages and cultural conventions is a key part of Metraly’s evolution.  Even though the platform is self‑hosted, customers may operate globally across different regions, currencies and scripts.  To ensure the UI scales gracefully, design and build with internationalisation (i18n) in mind from the outset.

## Design considerations

* **Flexible layouts.** Avoid hard‑coding widths or assuming that labels will always be a certain length.  Some languages (e.g. German) tend to produce longer strings, while others (e.g. Japanese) can be more compact.  Use fluid containers and allow text to wrap when necessary.
* **No embedded text in graphics.** Do not bake language‑specific text into icons or illustrations.  All text should come from a translation file so it can be replaced without editing images.
* **Clear iconography.** Use universally recognised icons to reduce reliance on text.  However, never rely solely on icons to convey meaning; pair them with translatable labels or tooltips.
* **Cultural nuance.** Be mindful of symbols, colours and metaphors that may not translate across cultures.  For example, red can signify danger in some contexts but luck in others.  Use the semantic colour tokens (e.g. `--color‑error`, `--color‑success`) consistently so that localisation teams can adjust palette values if needed.
* **Right‑to‑left (RTL) support.** Plan for layouts to mirror when switching to RTL languages like Arabic or Hebrew.  Use CSS logical properties (`margin‑inline‑start`, `padding‑inline‑end`) instead of physical ones (`margin‑left`, `padding‑right`), and test components in both directions.

## Implementation guidelines

* **Use a translation library.** Employ a robust i18n library such as [i18next](https://www.i18next.com/) or [react‑intl](https://formatjs.io/docs/react-intl/) to manage translation strings.  These tools provide context‑aware formatting for dates, times, numbers and plurals.
* **Keyed messages.** Reference all user‑facing strings via keys (`t('dashboard.noData')`) rather than inline text.  Keep keys descriptive and organised by domain (e.g. `charts.area.title`).
* **Date and number formatting.** Avoid hard‑coding locale formats.  Use the translation library’s helpers or the JavaScript `Intl` APIs to format dates, times, currencies and numbers based on the active locale.
* **Locale switching.** Provide a language selector component so users can choose their preferred language.  Persist the selection in local storage or a cookie and reload translations accordingly.
* **Testing across locales.** Include automated and manual tests that render key screens in multiple languages, including at least one RTL locale.  Watch for clipping, misaligned icons and truncated strings.

By baking internationalisation into the design system, Metraly can serve a broader audience while maintaining a consistent, accessible experience across cultures.
